ソフトウェアには誕生→開発・運用→廃止という一生がある。出発点は需要の発生で、これを要件定義書にまとめ「実現する機能／しない機能」を明確化する。設計でコーディング可能な仕様書とWBSに落とし込み、内製または外部委託で制作する。テストは「バグは必ずある」前提で早期発見に努めるが、工数の限界から全ては不可能。運用・保守では故障率がバスタブ曲線を描く。 プロジェクトは有期性と独自性を持ち、目標から逆算するバックキャスティングで進める。評価軸は品質・コスト・納期(QCD)で、FP法やバグ発生率で規模・品質を測る。開発プロセスにはウォーターフォール、スパイラル、反復型、アジャイル（スクラム・カンバン・XP等）があり、いずれも一長一短。選定は規模・重要度・人・変化・文化の5視点で判断し、YAGNIやCRACKが指針となる。 WBSはプロジェクトを成果物中心に階層分解した構成図で、スコープと作業を明確にする。トップダウンかつMECE（もれなくダブりなく）で作成する。

この開発プロセスを実際に回す基盤がバージョン管理であり、Gitはその分散型実装である。作業ツリー・ステージングエリア(index)・ローカルリポジトリ・リモートリポジトリの層を持ち、initで初期化、status/diff/logで状態・差分・履歴を確認、addでステージしcommitで履歴に確定する。修正系はcommit --amend（直前コミットの修正）、checkout（ファイルや過去コミットの復元）、reset（コミットのリセット）、revert（変更を打ち消すコミットを積む）、rm（ファイルとindexの削除）。branchで分岐しcheckoutで切替えmergeで統合する（--ff-onlyはfast-forward可能な場合のみ）。clone/pull/push/remoteでリモートと同期し、複数人が同一の履歴を共有する。
GitHubはリモートリポジトリのホスティングサービスであり、OSSの主要ポータルでもある。他ユーザーのリポジトリはcloneで複製でき、templateを使えば自分のリモートリポジトリとしていくつでも派生させられる。issuesは問題・課題・バグ・機能追加・質問を挙げる機能（Redmineのticketに相当）、projectsはそれらをテーブルやカンバンで管理する機能で、前述のドキュメンテーションと進捗管理がそのままツール化されている。開発環境自体もCodespacesによりブラウザ上のVSCodeとして起動でき、文書はマークダウン(.md)で書き、pandocでtex/pdf/docx/htmlへ変換できる。

GitHubフローは、mainを常にリリース可能な状態に保ち、作業はfeatureブランチ(dev)で行い、pushしてpull requestを出し、レビューを経てmainへmergeするという流れである。リリース時にタグ(1.0.0→1.1.0→1.2.0)を打ち、緊急の不具合はhotfixコミットとして1.1.1のように取り込む。ローカルブランチに上流がない場合は`git push --set-upstream origin <branch>`で対応づける。小さな変更を頻繁に統合するという反復型・アジャイル的な作法を、ツール側から促す仕組みといえる。
この流れを自動化したものがCI/CDである。CI(Continuous Integration)はコード変更を共有リポジトリへ頻繁に統合し、ビルドとテストを定期的に自動実行することでバグの早期発見・修正を可能にする。CD(Continuous Delivery)はテスト済みのコードをテスト環境・本番環境へ自動デプロイし、手動操作の必要性を排してユーザーフィードバックを迅速に反映する。パイプラインはソース→ビルド→テスト→デプロイ→検証→モニタリングの6段階で、失敗や問題検出があればソースへ戻る循環構造をとる。「バグは必ずあり早期発見が要だが全数テストは不可能」という制約に対する工学的な回答がこれであり、QCDのうちコストと納期を圧迫せずに品質を維持する手段になっている。
GitHub Actionsはその実装で、.github/workflows/にYAMLでワークフローを定義する。YAMLはXMLやJSONより簡潔な構文で、スペースによるインデント（タブ不可）で階層を、コロンでキーと値を、ハイフンでリストを表す。ワークフローはイベント・ジョブ・ステップから成り、onでトリガー（push、pull request、スケジュール等）を、jobsで並列実行可能なジョブを、stepsでシェルコマンド(run)や既製アクション(uses)の実行を指定する。実行環境runs-onはGitHub提供のホステッドランナー（ubuntu-latest等）とセルフホステッドランナーから選べ、actions/checkoutのような既製アクションも再利用できる。

演習で扱ったflake8はPythonの静的解析ツール（リンター）で、PEP8に照らした記法・スタイル上の逸脱を機械的に指摘する。flake8.ymlではon: pull_requestのpathsに'**.py'を指定して.pyを含むプルリクエストを契機とし、syntax-checkジョブをubuntu-latest上で起動、actions/checkout@v4でコードを取得、actions/setup-python@v5でPython 3系をセットアップ、pipでflake8を導入したうえで"flake8 . > flake8_report.txt || true"により解析結果をファイルへ出力し、actions/upload-artifact@v4でflake8_report.txtを成果物としてアップロードする。|| trueがあるため指摘が出てもジョブ自体は失敗せず、結果はActionsタブから確認できる。人手のレビューに依存していた品質チェックの一部を機械へ委譲することで、テスト工数の限界の下でも形式的な欠陥だけは毎回確実に検出できる——これがライフサイクルの「開発・運用」段階を支える実務的な基盤である。

他にGitHub Pagesはリポジトリの内容をそのままWebページとして公開する機能で、mainへのpushをトリガーにビルドと公開が自動で走る。「mainにmergeすれば成果物がそのまま本番環境へ届く」という構図であり、ワークフローを自前で書かずとも成立しているCDの最小実例といえる。ページ生成には静的サイトジェネレータのJekyllが用いられ、マークダウンで書いたファイルがテーマに従って自動的にHTML化されるため、書き手はコンテンツの記述だけに専念できる。
